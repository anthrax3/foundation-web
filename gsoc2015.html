The OpenBSD Foundation Ideas List - GSOC 2015

The OpenBSD Foundation gathered these projects and mentors together to support its successful application to be a mentoring organization for Google Summer of Code 2015.

As a Student, you can consider these ideas, but be sure to read up on How to Apply For Google Summer Of Code .

  • FIB route aggregation for OpenBGPd.
  • Design and implement network configuration daemon.
  • Implement BFD support as described in RFC 5880 and RFC 5881.
  • Add support for the NVGRE protocol.
  • Asynchronous USB transfer submission from userland.
  • Proper WebRTC integration in Mozilla.
  • Add sdmmc stack to sys/lib/libsa.
  • Infrastructure for integrating OpenBSD Ports and modules.
  • Unit Test, Measure and Replace LibreSSL data structures.
  • Implement a wscons version of libinput.
  • Enable OpenBSD to boot as guest on Xen.
  • Implement a simple KMS driver to the kernel.
  • Implement a tool to dump USB device descriptors.
  • Implement xHCI isochronous transfer support.
  • Project: FIB route aggregation for OpenBGPd.

    Brief explanation: With the default-free zone reaching 500,000 prefixes it is desirable to aggregate these prefixes by next-hop. The aggregates can then be inserted into the forwarding information base (FIB, i.e. the kernel routing table). This project involves designing and implementing an efficient algorithm (both in time and space) to calculate aggregates while there are constant updates / withdraws from BGP peers, some sort of rationale why that algorithm is correct, a regression test for the algorithm and finally implementing it in OpenBGPd.

    Expected results: Aggregated routes in the kernel routing table.

    Knowledge prerequisite: Strong understanding of C, Classless Inter-Domain Routing (CIDR), Border Gateway Protocol (BGP).

    Mentor: Henning Brauer hb-gsoc@bsws.de, Florian Obser florian@openbsd.org,

  • Project: Design and implement network configuration daemon.

    Brief explanation: Gathering the disparate network configuration mechanisms into one daemon that can provide coherence and a single point of control would greatly enhance the usefulness of OpenBSD in complex networking setups. The current mechanisms that are candidates for assimilation include dhcp, dhcp6. rtsol, and ifconfig. Similar efforts are underway on many os projects, which will serve as useful sources of ideas.

    Expected results: A single daemon subsuming at least two previous configuration mechanisms with a clear design for further assimilation.

    Knowledge prerequisite: C, kernal networking interfaces, dhcp.

    Mentor: Henning Brauer hb-gsoc@bsws.de

  • Project: Implement BFD support as described in RFC 5880 and RFC 5881.

    Brief explanation: Bidirectional Forwarding Detection is a protocol to actively detect faults in the forwarding path between two routers. In OpenBSD BFD will extend the current link-state tracking to do end to end test between two hosts. This project involves writing a basic BFD implementation which in a second step will be integrated into the network stack. Finally interoperability with various other systems needs to be tested.

    Expected results: A working BFD implementation for OpenBSD.

    Knowledge prerequisite: Strong understanding of C, Networking.

    Mentor: Henning Brauer hb-gsoc@bsws.de

  • Project: Add support for the NVGRE protocol.

    Brief explanation: Support for virtual overlay networks with the VXLAN protocol has been introduced in OpenBSD 5.5 with the vxlan(4) driver. An alternative approach is the NVGRE protocol that that is currently described in the IETF draft "draft-sridharan-virtualization-nvgre-07". It is an extension to the GRE protocol and can be added to OpenBSD's gre(4) driver. It would need support for the GRE key, the NVGRE headers, integration into bridge(4) and ifconfig(8) using the same interfaces and semantics that have been created for vxlan(4). Additional interoperability testing with Microsoft's Windows Server 2012 or newer would be required.

    Expected results: A patch for gre(4) to support the GRE key and NVGRE.

    Knowledge prerequisite: C, kernel networking interfaces, Ethernet, IP, Windows Server configuration.

    Mentor: Reyk Floeter reyk@openbsd.org

  • Project: Asynchronous USB transfer submission from userland.

    Brief explanation: Extend the existing interface exposed by ugen(4) and usb(4) and used mainly though libusb, or design a new one, to be able to submit USB transfers asynchronously.

    Expected results: A working interface supporting at least control, bulk and interrupt transfers.

    Knowledge prequisite: C, USB, device drivers.

    Mentor: Martin Pieuchot mpi@openbsd.org

  • Project: Proper WebRTC integration in Mozilla.

    Brief explanation: The WebRTC API within Firefox is not yet fully functional nor stable on OpenBSD. Sound integration not finished, testing coverage very low.

    Expected results: WebRTC working & enabled by default within Firefox in OpenBSD's ports-tree. Patches properly integrated upstream.

    Knowledge prerequisite: C, IPC.

    Mentor: Landry Breuil landry@openbsd.org

  • Project: Add sdmmc stack to sys/lib/libsa.

    Brief explanation: One of most problematic missing part for having a working boot block on armv7 is that there is no minimal sdmmc stack code in sys/lib/libsa.

    Expected results: A working sdmmc stack in sys/lib/libsa.

    Knowledge prequisite: C, device drivers, ARM.

    Mentor: Miod Vallat miod@openbsd.org

  • Project: Infrastructure for integrating OpenBSD Ports and module installation frameworks.

    Brief explanation: OpenBSD Ports contain many small ports for Ruby gems, CPAN modules and so on. Those ports are mostly simple "wrappers" that rely on corresponding port modules. There are a few problems with this design: 1) in OpenBSD Ports, it's needed to manually duplicate version requirements from upstream package; 2) maintaining a full port structure for each small upstream package consumes too much time from maintainers.

    Expected results: Fully-automated process of creating and updating ports for upstream modules packages.

    Knowledge prerequisite: C, perl.

    Mentor: Vadim Zhukov zhuk@openbsd.org, Andrew Fresh afresh1@openbsd.org

  • Project: Unit Test, Measure and Replace LibreSSL data structures

    Brief explanation: LibreSSL has inherited a number of files to implement basic data structures (e.g. hashing, lists) which are then used internally to the library. The implementation of many of these data structures could benefit from review, and possible reimplementation to a more safe and efficient standard.

    This project would be to write unit tests and measurements to both verify the correctness of the internal data structure implementations, and measure them for benchmarking purposes. Once that is done an effort can be made to selectively re-implement some of these data structures - probably in coordination with LibreSSL developers.

    Expected results: Unit tests and measurements for internal LibreSSL data structures.

    Knowledge prerequisite: Testing, measurement, a strong background in C and data structures - ability for independent research and measurement.

    Mentor: Bob Beck beck@obtuse.com, Joel Sing jsing@openbsd.org

  • Project: Implement a wscons version of libinput.

    Brief explanation: The wscons version of libinput will open the road for enabling multi-touch support in X and a future Wayland port.

    Expected results: A working wscons version of libinput.

    Knowledge prerequisite: X, C.

    Mentor: Matthieu Herrb matthieu@herrb.eu

  • Project: Xen guest support for OpenBSD

    Brief explanation: The Xen hypervisor exposes a virtual guest ABI for x86_32, x86_64 and ARMv7 architectures. Several portions of this ABI are architecture-independent, such as device drivers for the Xen device drivers and XenStore. Support can be ported from NetBSD and FreeBSD, which have had support for a few years now.

    Expected results: Boot OpenBSD as an accelerated HVM guest or a native PV guest on Xen.

    Knowledge prerequisite: Strong understanding of C and/or operating systems.

    Mentor: Anil Madhavapeddy anil@recoil.org

  • Project: Implement a simple KMS driver to the kernel.

    Brief explanation: The X server is moving away from the legacy model where drivers had direct access to the hardware to set the video modes and handle acceleration. This project is about creating more KMS drivers beyond the existing intel and radeon ones. It can target a relative simple driver like Cirrus (that's useful for people running under Qemu). This would make it possible to use the xf86-video-modesettings driver to run X with allowaperture=0 (ie without privileges) on these systems.

    Expected result: An implementation of a kernel KMS driver for one simple graphics card, providing the basic graphics mode settings operations needed to run X with the xf86-video-modesettings driver.

    Knowledge prerequisites: C, device drivers. Notions about the X.Org driver architecture.

    Mentor: Matthieu Herrb matthieu@herrb.eu

  • Project: Implement a tool to dump USB device descriptors.

    Brief explanation: Extend usbdevs(8) or write a new tool to dump the device descriptors of connected USB devices, something like "lsusb -v".

    Expected result: A working tool able to at least dump Device, Configuration, Interface and Endpoint descriptors.

    Knowledge prerequisites: C, USB, ioctls.

    Mentor: Martin Pieuchot mpi@openbsd.org

  • Project: Implement xHCI isochronous transfer support.

    Brief explanation: xhci(4) can currently submit control, bulk and interrupt transfers. Supporting isochronous transfers would make it possible to use devices like uvideo(4) or uaudio(4) behind it.

    Expected result: A patch for xhci(4), and if necessary to the USB stack, to support isochronous transfers.

    Knowledge prerequisites: C, USB, xHCI, device drivers.

    Mentor: Martin Pieuchot mpi@openbsd.org
